# Phase 00: Daily Stock Analysis 数据源集成可行性调研
**Researched:** 2025-04-15
**Domain:** 市场数据多源调度集成
**Confidence:** HIGH

## 摘要
本次调研旨在评估将 `daily_stock_analysis` 项目的多数据源拉取能力集成到 `control-plane` 现有同步任务体系中的可行性。通过对两个系统的架构分析、接口对比、差异梳理，确认集成方案具备技术可行性，仅需少量适配工作即可实现能力复用。

**核心结论：** 集成方案完全可行，预计开发周期 1-2 周，可复用 `daily_stock_analysis` 成熟的多数据源管理能力，提升现有同步系统的容错性和扩展性。

## 现有控制平面同步架构分析
### 核心架构特性
1. **任务调度**：基于 `node-cron` 实现定时任务，支持灵活的 cron 表达式配置
2. **数据源管理**：手动实现多数据源切换，缺乏统一的熔断降级机制
3. **执行模型**：TypeScript 主进程调用 Python 子进程拉取数据
4. **事务保证**：临时表 + 事务原子写入，避免脏数据
5. **数据校验**：内置数量校验，少于阈值自动回滚
6. **日志追踪**：全链路任务执行日志，便于问题排查

### 现有同步流程
```
任务触发 → Python 脚本执行 → 多数据源尝试拉取 → 数据校验 → 临时表写入 → 正式表替换 → 日志记录
```

### 存在的问题
1. **多源管理能力薄弱**：每个 Python 脚本独立实现数据源 fallback，代码重复率高
2. **缺乏熔断机制**：单个数据源频繁失败不会自动熔断，可能导致上游服务不可用
3. **扩展成本高**：新增数据源需要修改所有相关脚本，维护成本高
4. **错误处理不统一**：各个脚本独立处理错误，缺乏统一的重试、降级策略

## Daily Stock Analysis 数据源架构分析
### 核心架构特性
1. **统一抽象层**：基于 `BaseFetcher` 抽象类，所有数据源实现统一接口
2. **智能调度器**：`DataFetcherManager` 实现自动优先级调度、失败切换、熔断保护
3. **丰富的数据源支持**：内置支持 AkShare、Efinance、Tushare、Pytdx、YFinance、LongBridge 等 6+ 数据源
4. **多市场适配**：支持 A股、港股、美股、ETF 等不同市场标的
5. **批量预取能力**：支持全量预取 + 缓存，大幅提升批量拉取效率
6. **统一数据标准化**：所有数据源返回格式统一，上层业务无需关注数据源差异

### 核心组件
| 组件 | 职责 | 优势 |
|------|------|------|
| `BaseFetcher` | 数据源抽象基类 | 统一接口，便于扩展新数据源 |
| `DataFetcherManager` | 数据源管理器 | 自动优先级调度、失败切换、熔断 |
| `realtiem_types` | 数据结构定义 | 统一数据格式，标准化返回 |
| `fundamental_adapter` | 基本面数据适配 | 多源基本面数据聚合 |

### 数据源优先级配置
```
默认优先级：
1. Efinance (优先级最高)
2. Akshare

特殊规则：

- 美股优先使用 LongBridge/Yfinance
- 港股优先使用 LongBridge/AkShare
```

## 集成适配点分析
### 现有控制平面同步任务接口规范
每个同步任务实现以下结构：
```typescript
// 1. 任务入口函数
export async function syncXxxData() {
  // 2. 日志记录
  const taskId = await logTaskStart('sync_xxx_data')

  try {
    // 3. 调用 Python 脚本拉取数据
    const data = await runPythonScript('fetch_xxx_data.py', [], timeout)

    // 4. 数据校验
    if (data.length < threshold) {
      throw new Error('Data incomplete')
    }

    // 5. 事务写入
    const client = await pool.connect()
    try {
      await client.query('BEGIN')
      // 临时表写入 + 正式表替换
      await client.query('COMMIT')
    } catch (e) {
      await client.query('ROLLBACK')
      throw e
    }

    // 6. 成功日志
    await logTaskSuccess(taskId, data.length, 'datasource')

  } catch (e) {
    // 7. 失败处理
    await logTaskFailed(taskId, errorMsg, syncCount)
    throw e
  }
}
```

### Python 脚本调用协议
现有 Python 脚本遵循以下规范：
1. **输入**：通过命令行参数传递参数
2. **输出**：成功时输出 JSON 数组到 stdout，失败时输出错误信息到 stderr 并以非 0 状态码退出
3. **数据格式**：JSON 数组，每个元素为标准化的业务对象
4. **降级逻辑**：脚本内部实现多数据源 fallback，无需上层干预

### 集成适配方案
#### 方案一：Python 侧适配（推荐）
将 `daily_stock_analysis` 的数据源能力封装为符合现有协议的 Python 脚本：
1. 复用现有 `DataFetcherManager` 能力，实现多数据源自动切换
2. 输出格式与现有脚本完全一致，无需修改 TypeScript 侧代码
3. 继承现有熔断、重试、降级机制

**适配工作量：** 低，每个同步任务仅需实现 20-30 行封装代码

#### 方案二：TypeScript 侧集成
通过 Node.js 调用 Python 模块直接使用数据源能力：
1. 封装 Node.js 层调用 `DataFetcherManager` 的接口
2. 适配现有任务调度框架
3. 保留 TypeScript 侧的事务写入和日志能力

**适配工作量：** 中，需要实现跨语言调用适配层

### 差异点对比与解决方案
| 差异点 | 现有控制平面 | Daily Stock Analysis | 适配方案 |
|---------|-------------|----------------------|---------|
| 开发语言 | TypeScript + Python | Python 为主 | 保留 Python 侧实现，封装为符合现有协议的脚本 |
| 数据源管理 | 脚本内手动实现 | 统一 `DataFetcherManager` 调度 | 复用 `DataFetcherManager` 替换原有手动 fallback 逻辑 |
| 熔断机制 | 无 | 内置熔断、重试、降级 | 直接复用，无需额外开发 |
| 数据格式 | 自定义内部格式 | 标准化统一格式 | 新增适配层转换为现有格式，或直接升级统一格式 |
| 配置管理 | 环境变量 + 配置文件 | 环境变量 + 配置中心 | 统一环境变量命名，保持配置兼容性 |
| 日志体系 | 自定义日志格式 | 结构化日志 | 适配日志格式，统一输出到现有日志系统 |
