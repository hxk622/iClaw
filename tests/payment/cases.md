# Payment E2E Cases

适用范围：
- `admin-web` 可视化支付配置
- `desktop` 充值中心 / 扫码支付流程
- `control-plane` 真实下单、订单状态、平台级支付网关读取

目标：
- 覆盖“平台级 + OEM 继承”支付主链路
- 覆盖用户真实操作路径，而不是只测 API
- 每个 case 都能配合 CDP + 截图做回归确认

建议执行前置：
- `admin-web` 运行在 `http://127.0.0.1:1479`
- `desktop web shell` 运行在 `http://127.0.0.1:1520`
- `control-plane` 运行在 `http://127.0.0.1:2130`
- Chrome CDP 端口已开启，统一走根目录 `tests/` 方案

## Case 01: 平台支付网关可视化配置读取

ID:
- `PAY-E2E-001`

目标:
- 验证 `admin-web -> 支付中心 -> 账户配置 -> 平台` 页面展示的是数据库真实值

步骤:
1. 打开 `admin-web`
2. 登录管理员账号
3. 进入 `支付中心 -> 账户配置`
4. 保持在 `平台` tab
5. 读取 `partner_id / gateway / key`
6. 截图保存

断言:
- 页面存在 `payment-gateway-form`
- 三个字段都可见
- 三个字段值与当前平台配置一致
- 页面显示来源为 `admin-web` 或 `admin`

截图点:
- 平台支付网关表单完整截图

备注:
- 这是最基础的“配置真值读取”校验

截图点:

## Case 05: 二维码支付卡片一屏可见且卡片本身不滚动

ID:
- `PAY-E2E-005`

目标:
- 验证支付卡片布局满足“一眼看全二维码”，且卡片本身无内层滚动 bug

步骤:
1. 在 desktop 打开支付二维码页
2. 检查二维码、支付方式 logo、套餐摘要、主按钮是否都在首屏内
3. 用 CDP 读取支付卡片 `scrollHeight/clientHeight/overflow`
4. 截图保存

断言:
- 支付卡片区域无垂直滚动条
- `scrollHeight <= clientHeight` 或 overflow 不为可滚动状态
- 二维码完整可见，无裁切
- 页面主信息不需要在卡片内部滚动才能看全

截图点:
- 支付页完整截图
- 如有必要补一张二维码局部截图

## Case 06: 过期二维码出现遮罩与刷新按钮

ID:
- `PAY-E2E-006`

目标:
- 验证订单过期后，二维码出现遮罩，并可通过刷新按钮重建订单

步骤:
1. 创建一个支付订单并进入二维码页
2. 使用测试手段让订单进入 `expired`
3. 等待前端轮询或主动刷新状态
4. 观察二维码遮罩与刷新按钮
5. 点击刷新
6. 校验新的订单号或新的支付链接
7. 截图保存

断言:
- 过期后出现遮罩层
- 遮罩层含刷新按钮
- 点击刷新后生成新订单
- 新订单 `payment_url` 与旧订单不同

截图点:
- 过期前二维码
- 过期后遮罩
- 刷新后的新二维码

## 建议脚本拆分

建议后续按下面的文件粒度落 CDP 脚本：
- `tests/payment/admin-gateway-read.test.mjs`
- `tests/payment/recharge-package-flow.test.mjs`
- `tests/payment/qr-layout.test.mjs`
- `tests/payment/expired-refresh.test.mjs`
- `tests/payment/oem-method-inheritance.test.mjs`
- `tests/payment/admin-order-linkage.test.mjs`

这样拆的原因：
- 每个脚本聚焦一条主链路
- 失败定位更快
- 便于你单独点跑某一个脚本
