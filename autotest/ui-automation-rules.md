# UI 自动化规则

本文件用于约束 UI 自动化测试、录制脚本、回归执行与问题复现流程，确保脚本稳定、可维护、可复用。

## 1. 基础目标

- UI 自动化优先覆盖核心主流程、关键配置流程、登录鉴权流程、支付/授权/保存等高风险路径。
- 自动化脚本必须以“稳定复现业务行为”为目标，禁止为了通过用例而依赖脆弱选择器或临时等待。
- 所有脚本应具备可重复执行能力，在本地与 CI 环境下结果应尽量一致。

## 2. 环境约束

- 前端地址固定为 `http://127.0.0.1:1520`。
- OAuth 回调页固定为 `http://127.0.0.1:1520/oauth-callback.html`。
- 控制平面后端为 `http://127.0.0.1:2130`。
- OpenClaw 本地 API/runtime 为 `http://127.0.0.1:2126`。
- 不允许在自动化脚本中随意改动端口、域名、回调地址；如需调整，必须显式说明原因并同步更新所有依赖配置。

## 3. 用例设计规则

- 一个用例只验证一个清晰业务目标，避免在单个脚本中串联过多无关路径。
- 优先覆盖用户真实操作路径，不以实现细节为中心设计断言。
- 每个场景至少包含：
  - 前置条件
  - 操作步骤
  - 预期结果
  - 清理方式（如有脏数据）
- 新增功能必须补充正常路径用例；有权限、状态切换、失败分支时，必须补充异常路径用例。
- 配置驱动 UI 不仅要验证页面显示，还要验证保存后的持久化结果及运行时实际生效结果。

## 4. 元素定位规则

- 优先使用稳定定位方式，推荐顺序：
  1. `data-testid` / `data-test`
  2. 可访问性语义定位（role、name、label）
  3. 稳定文本
  4. 结构化 CSS 选择器
- 禁止使用易碎定位方式：
  - 依赖随机 class 名
  - 依赖 DOM 层级序号
  - 依赖视觉位置
  - 依赖会频繁变化的文案片段
- 当页面缺少稳定标识时，应优先补充测试标识，而不是堆叠复杂选择器。

## 5. 等待与同步规则

- 禁止滥用固定等待，如 `sleep 1000`、`waitForTimeout(5000)` 作为主同步手段。
- 必须基于真实状态同步：
  - 元素出现/消失
  - 按钮可点击
  - 请求完成
  - 文本更新
  - 路由跳转完成
- 对异步保存、懒加载、弹窗、toast、表格刷新场景，必须显式等待最终可验证状态。

## 6. 断言规则

- 断言必须贴近业务结果，不只断言“页面没报错”。
- 推荐断言内容包括：
  - 页面关键文案出现
  - 表单值成功保存
  - 列表数据新增/更新/删除成功
  - 权限限制生效
  - 配置在运行时接口或公开 payload 中可见
- 对配置类页面，保存后至少做两层验证：
  1. 验证后台持久化记录正确
  2. 验证前台运行态/公共接口实际生效

## 7. 数据管理规则

- 测试数据必须可追踪、可清理、可重复创建。
- 避免依赖共享脏数据或人工预置数据。
- 用例间默认相互独立，禁止隐式依赖执行顺序。
- 如必须复用账号、组织、应用实例，应约定命名规范并控制数据生命周期。

## 8. 登录与权限规则

- 登录态应优先通过稳定方式建立，如接口注入、预置 session、受控测试账号。
- 涉及 OAuth、鉴权跳转、权限控制时，必须覆盖：
  - 未登录访问
  - 登录后跳回原路径
  - 无权限展示或禁用逻辑
  - token/session 失效后的处理

## 9. 稳定性规则

- 脚本必须可重复运行，不允许首轮通过、次轮失败。
- 对弹窗、遮罩、动画、延迟渲染等不稳定因素，应通过等待条件、关闭动画或稳定标识处理。
- 对易波动接口应使用受控测试环境或稳定 mock，不得让第三方波动直接破坏核心回归。
- 遇到偶发失败时，必须先修复根因，禁止长期依赖重试掩盖问题。

## 10. 失败产物规则

- 用例失败时必须保留可排查信息，至少包含以下之一：
  - 截图
  - 页面源码快照
  - 控制台错误
  - 网络请求日志
  - 视频或 trace
- 报错信息要能直接说明：
  - 哪一步失败
  - 预期是什么
  - 实际是什么

## 11. 维护规则

- 页面结构变化后，应优先修正公共页面对象、公共选择器和公共操作方法，避免逐个脚本打补丁。
- 公共流程如登录、创建应用、切换组织、打开设置页，应抽为可复用能力。
- 自动化脚本命名必须表达业务意图，禁止使用 `test1`、`demo`、`final` 等无语义名称。
- 删除或重构功能时，应同步清理失效自动化用例，避免保留无意义历史脚本。

## 12. 评审规则

- 每个新增 UI 自动化用例都应回答以下问题：
  - 这个场景是否真的是高价值路径
  - 是否用了稳定定位
  - 是否避免了固定等待
  - 断言是否验证到了业务结果
  - 数据是否可重复构造和清理
  - 失败后是否容易排查

## 13. 推荐执行口径

- 冒烟：覆盖登录、首页、关键导航、核心创建流程、核心保存流程。
- 回归：覆盖 OEM 配置、生效链路、权限链路、运行态展示链路。
- 发版前：至少执行冒烟 + 高风险回归场景。

## 14. 一句话原则

UI 自动化不是“把点击录下来”，而是用稳定、可验证、可维护的方式证明关键业务流程真的可用。
