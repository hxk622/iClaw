# Admin Web Interaction Standard

更新时间：2026-04-11

## 目标

`admin-web` 的 React 重构不只是把旧页面搬过去，还要统一后台交互模型，避免同一套运营后台里同时出现多种互相冲突的编辑方式。

## 总原则

所有后台运营页优先遵循以下顺序：

1. 先按 `tab` 切作用域或大类。
2. 每个 `tab` 内优先显示统计区域。
3. 若当前 `tab` 需要管理多个对象，采用“列表卡片 + 详情抽屉”组合。
4. 详情编辑始终在当前页完成，不跳 legacy，不进入二级页面。

## 标准结构

### 1. Tab 结构

适用场景：

- 平台 / OEM
- release / binding / history
- chat provider / memory embedding

要求：

- `tab` 只承担“切作用域 / 切配置视图”
- 不在 `tab` 里再嵌套另一层页面跳转

### 2. 统计区域

每个重要 `tab` 内都应该有一块统计区域，至少回答：

- 当前对象总数
- 已启用 / 已绑定数量
- 当前选中对象
- 当前作用域状态

实现形式：

- 优先使用 `fig-meta-cards`

### 3. 列表卡片 + 详情抽屉

适用场景：

- 左菜单栏
- 平台能力管理
- 云目录管理
- 任何“同类对象列表 + 当前对象编辑”的页面

标准布局：

- 左侧：列表卡片区
  - 支持筛选
  - 支持排序
  - 支持选择当前对象
- 右侧：详情抽屉区
  - 固定当前对象摘要
  - 展示该对象完整编辑表单
  - 保持 sticky，避免长列表场景中编辑区丢失

要求：

- 列表区负责“找对象 / 选对象 / 排对象”
- 抽屉区负责“看详情 / 改配置 / 执行动作”
- 不再使用“左列表 + 右侧普通平铺详情页”的模糊模式

### 4. 动作闭环

凡旧版后台已有的运营动作，在 React 版里必须保留：

- 新建
- 保存
- 删除
- 启用 / 停用
- 测试连接
- 发布 / 恢复
- 导出

动作位置要求：

- 当前对象动作放在详情抽屉顶部或底部 action 区
- 全局动作放在页面 header 或统计区旁边

## 当前已落地

- 模型中心：`tab` 工作台
- 品牌详情左菜单栏：`列表卡片 + 右侧详情抽屉`
- 平台 Skill / MCP：当前对象工作台
- 云 MCP：当前对象工作台

## 后续执行要求

- 新增后台页面时，不再临时发明新的交互模式
- 发现旧页面仍使用过时的 master-detail 或纯列表模式时，优先向本规范收敛
