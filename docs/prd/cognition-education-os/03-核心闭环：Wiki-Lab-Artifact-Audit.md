# 03. 核心闭环：Wiki-Lab-Artifact-Audit

## 为什么平台核心不是 Memory

平台不是一个“记住更多信息”的系统，而是一个“把认知变成规则，再把规则放进压力环境里审计”的系统。

所以核心不应只是 memory，而应是一个闭环：

```text
Wiki -> Lab -> Artifact -> Audit -> Rule Repair
```

## 1. Wiki

作用：
- 把知识、概念、因果关系、学习日志、用户认知盲点组织成结构化知识层
- 为实验室和审计提供可调取的上下文

## 2. Lab

作用：
- 在压力环境中暴露“知行不一”
- 不是测记忆，而是测系统在极端场景下是否仍能工作

## 3. Artifact

作用：
- 让用户把规则、边界、判断依据外化成可审计对象
- Artifact 是认知系统的“显式版本”

## 4. Audit

作用：
- 对比“用户之前写下的规则”和“用户在压力中做出的动作”
- 暴露裂缝
- 强制触发修正，而不是停留在记录层

## 5. Rule Repair

平台真正完成闭环的标志不是“用户学过了”，而是：
- 写下了规则
- 违背了规则
- 被系统审计出来
- 被迫修改规则或承认自己的真实模型

## 平台成功的最小闭环

1. 用户完成某一模块学习
2. 相关知识进入本地 wiki
3. 系统基于漏洞生成压力实验
4. 用户在压力下做出决策
5. 系统要求写入或更新 artifact
6. 系统执行 logic audit
7. 形成前后版本差异与修正轨迹
