# 分发原则与防坑脚本说明

## 一句话原则

- **龙虾商店 = 全集**
- **智能投资专家 = 金融视图**

也就是：

1. 新蒸馏出来的专家，**默认永远进入龙虾商店**
2. 只要是**金融相关**专家，同时进入**智能投资专家**
3. 所以金融专家的 `surface` 不是 `investment-experts`，而是 **`both`**
4. 非金融专家的 `surface` 应为 **`lobster-store`**

## 为什么这样定

### 1. 龙虾商店是平台全集

龙虾商店承担平台总商店职责，不能只放一部分。

### 2. 智能投资专家是金融视图，不是独立总库

智能投资专家只是对龙虾商店中的金融 expert 做一层聚合与筛选。

### 3. 这样才不会再出现分发混乱

之前踩过的坑，本质都是因为没有把“全集”和“金融子集”分开定义。

---

## 已踩过的坑

### 坑 1：把金融 expert 只发到 `investment-experts`

错误做法：

- `metadata.surface = investment-experts`

后果：

- 智能投资专家里能看见
- 龙虾商店里看不见
- 与“龙虾商店是全集”的原则冲突

正确做法：

- 金融 expert 一律 `metadata.surface = both`

---

### 坑 2：头像使用外链 URL，导致 UI 显示不稳定

错误做法：

- `avatar_url` 直接填 `unsplash` / 外部 CDN / 临时图片地址

后果：

- 龙虾商店和智能投资专家页的显示一致性差
- 有时能显示，有时空白
- 很难判断是前端问题还是资源问题

正确做法：

- 优先使用 `.tmp/avatars/<slug>.data-url.txt`
- 发布前自动内联成 `data:image/...`

---

### 坑 3：发布后 catalog 还是旧数据

原因：

- control-plane 有缓存
- 页面端也有 catalog cache

后果：

- 数据库已经是新值
- 页面仍拿到旧 avatar / 旧 surface / 旧排序

正确做法：

1. 发布前先规范化输入
2. 发布后必要时重启本地 control-plane
3. 页面刷新时强制拉最新 catalog

---

### 坑 4：只看未登录截图验收

错误做法：

- 用游客态截图证明“已经上架”

后果：

- 与真实用户态不一致
- 不能作为有效验收依据

正确做法：

- 必须使用**登录态**验收截图

---

## 现在沉淀下来的脚本

### 1. 规范化脚本

- `.codex/skills/investment-persona-distiller/scripts/normalize_investment_expert_candidates.mjs`

职责：

1. 自动把分发表达式统一成：
   - 金融 → `both`
   - 非金融 → `lobster-store`
2. 自动尝试把 `.tmp/avatars/<slug>.data-url.txt` 内联进 `avatar_url`
3. 可同步回写 `.tmp/nuwa-distill/<slug>/package/metadata.json`

### 2. 发布脚本

- `.codex/skills/investment-persona-distiller/scripts/publish_investment_expert_candidates.sh`

现在发布前会先自动调用规范化脚本，避免再手工漏改：

- `surface`
- `avatar_url`
- package metadata 同步

## 未来真人 IP 蒸馏的分发规则

对“巴菲特 / 段永平 / 芒格 / 达利欧 / 霍华德·马克斯 / 塔勒布”这类真人蒸馏 expert：

### 如果是金融人物

- `category = finance`
- `metadata.surface = both`
- 同时出现在：
  - 龙虾商店
  - 智能投资专家

### 如果未来是非金融人物

例如：

- 张一鸣
- 费曼
- MrBeast

则：

- 默认进入龙虾商店
- 不进入智能投资专家

---

## 结论

以后不要再把“龙虾商店”和“智能投资专家”当成两套平行库。

正确理解只有一句：

> **龙虾商店是全集，智能投资专家是金融视图。**
