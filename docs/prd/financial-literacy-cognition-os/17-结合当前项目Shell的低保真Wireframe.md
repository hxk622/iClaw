# 17. 结合当前项目 Shell 的低保真 Wireframe

## 目标

这份 wireframe 不是脱离仓库的“理想化新 App”，而是基于当前 iClaw 项目的实际壳层结构，定义财商教育产品如何嵌入现有 shell。

## 当前项目可复用的壳层事实

基于当前仓库，可确认的相关事实包括：

1. **桌面壳已存在**：`apps/desktop/` 是当前主应用壳。
2. **Home 官网已存在**：`home-web/` 是独立官网/落地页入口。
3. **OEM shell surfaces 已存在配置模型**：`admin-web/main.js` 已包含 `home-web`、`header`、`sidebar`、`input` 等 surface 配置概念。
4. **桌面 v0 左侧菜单当前仍偏占位**：`apps/desktop/README.md` 明确写明左侧菜单和设置相关能力在 v0 为静态占位。

因此，新的“金融财商认知操作系统”最合理的落地方式不是另起一套完全独立壳，而是：

> 在现有 shell 内新增一个“学习/认知系统”产品域，并通过 OEM shell 配置逐步把它从静态入口演进为正式模块。

---

## 建议的壳层分工

### 1. `home-web/`
**角色：** 市场入口 / 产品介绍 / 转化入口  
**承载：**
- 产品介绍
- 学习价值说明
- 课程与实验室介绍
- 注册/下载/进入学习系统入口

### 2. `apps/desktop/`
**角色：** 核心学习与互动壳  
**承载：**
- 学习首页
- 学习地图
- 认知实验室
- IPS 工作台
- 复盘记录

### 3. OEM shell surfaces
**角色：** 壳层装配与导航配置  
**承载：**
- 左侧菜单注入
- header 文案与入口
- 首页卡片入口
- 学习模块可见性与排序

---

## 推荐集成策略

建议采用两层集成：

### 第一层：产品域接入现有桌面壳
在左侧菜单中新增一级域：
- 学习

展开后可包含：
- 今日任务
- 学习地图
- 实验室
- IPS
- 复盘

### 第二层：OEM 化能力
后续把上述菜单与首页入口逐步纳入 OEM shell 配置，使不同品牌 app 可以：
- 是否启用学习系统
- 自定义学习入口名称
- 自定义首页 CTA
- 定义默认落地页

## Shell 级低保真 Wireframe

### A. 桌面壳：左侧导航级集成

```text
┌──────────────────────────────────────────────────────────────┐
│ iClaw / OEM App                                             │
├───────────────┬──────────────────────────────────────────────┤
│ Logo          │ Header                                       │
│               │  搜索 / 账户 / 进度摘要 / 全局状态             │
│ 对话          ├──────────────────────────────────────────────┤
│ 学习          │                                              │
│  ├ 今日任务    │                                              │
│  ├ 学习地图    │               主内容区域                      │
│  ├ 实验室      │                                              │
│  ├ IPS        │                                              │
│  └ 复盘       │                                              │
│ 任务中心       │                                              │
│ 记忆          │                                              │
│ 设置          │                                              │
└───────────────┴──────────────────────────────────────────────┘
```

### 设计说明
- “学习”成为和“对话”并列的一级产品域
- 学习域内部再拆分任务、课程、实验、IPS、复盘
- 这样既不破坏现有聊天壳，也能让学习系统成为正式产品区而不是弹窗式功能

---

### B. Home 官网：学习产品入口级集成

```text
┌──────────────────────────────────────────────────────────────┐
│ 顶部导航：产品 / 财商认知 / 下载 / 登录                      │
├──────────────────────────────────────────────────────────────┤
│ Hero                                                        │
│ “不是教你追热点，而是帮你建立金融认知操作系统”               │
│ [立即开始学习] [了解认知实验室]                              │
├──────────────────────────────────────────────────────────────┤
│ 产品三段说明                                                 │
│ 课程系统 | 认知实验室 | IPS 工作台                           │
├──────────────────────────────────────────────────────────────┤
│ 90 天学习路径预览                                            │
├──────────────────────────────────────────────────────────────┤
│ 下载 / 注册 / 进入桌面端                                     │
└──────────────────────────────────────────────────────────────┘
```

### 设计说明
- Home 承担“价值解释 + 转化入口”，不直接承担完整学习闭环
- 学习系统的深互动发生在桌面壳内更合理

---

## Header 级接入建议

当前 header 可增加一个轻量学习状态区：

```text
[当前阶段：模块 3 / 8] [IPS 完成度 62%] [待复盘 1]
```

作用：
- 把学习系统变成全局状态，而不是孤立模块
- 提醒用户当前在“认知安装过程”中的位置

---

## Sidebar 级接入建议

### 推荐菜单结构

```text
学习
├─ 今日任务
├─ 学习地图
├─ 认知实验室
├─ IPS 工作台
└─ 复盘记录
```

### 不建议的结构

```text
工具
├─ 财商学习
```

原因：
- 如果只把它放在“工具”下面，会被误解成边缘功能
- 该产品应是独立产品域，不是附属插件

---

## 输入框 / Chat 壳的关系

不建议 v1 直接把“学习系统”做成纯聊天代理。

### 原因
- 聊天交互适合问答，但不适合承载完整课程路径、实验室压力测试、IPS 工作台
- IPS 的核心是结构化沉淀，而不是对话漂流

### 但可保留的辅助关系
- 在课程页中嵌入“问问 AI 解释这个图”
- 在 IPS 页中嵌入“帮我检查这条规则是否模糊”
- 在实验室页中嵌入“解释我为什么在这里犯错”

结论：
> Chat 更适合作为学习系统的辅助手段，而不是主容器。

---

## v1 壳层落地建议

### Phase 1
- 在桌面左侧新增“学习”一级入口
- 在 Home 页增加学习产品介绍区
- 学习系统内部先做独立内容容器

### Phase 2
- 把学习入口、标题、首页 CTA 收敛到 OEM shell surfaces
- 支持不同 app 品牌配置默认落点

### Phase 3
- 把学习进度与 header / home shell 联动
- 做成完整 OEM 可配置产品域

---

## 壳层落地判断标准

如果一个方案：
- 不能进入现有桌面主导航
- 不能清晰区分 home 入口与 desktop 深度体验
- 不能利用 OEM shell 的 header/sidebar/home 配置模型

那它就还不是“结合当前项目结构”的落地方案。
