# 10. 风险、约束与后续工作流

## 产品约束

- 必须坚守教育型定位
- 不能滑向投顾、荐股、交易执行
- 个性化必须服务于认知升级，而非刺激交易
- 游戏化必须服务于“低成本犯错 + 高质量复盘”

## 主要风险

### 风险 1：内容太重，用户学不下去
**应对：** 路径要短、模块要高密度、用模拟带来动机，而不是无限拉长课程。

### 风险 2：模拟器像 gimmick，不像认知工具
**应对：** 每个场景必须绑定明确的认知错误与 IPS 修正规则。

### 风险 3：IPS 评价过于主观
**应对：** 在试点前先定义标准化 rubric，建立可复核的评审标准。

### 风险 4：产品边界失控，长成投资助手
**应对：** 所有需求评审都要先过“教育边界检查”。

### 风险 5：内容生产成本太高
**应对：** 先从窄而深的课程骨架和可复用的模拟模板开始。

## 后续工作流建议

### 如果进入产品定义/原型阶段
建议拆成 4 条线：
1. **产品/架构线**：定义系统边界、信息架构、数据与内容结构
2. **UX 线**：学习路径、实验室交互、IPS 写作体验
3. **内容线**：课程骨架、模型图解、推理题与场景脚本
4. **验证线**：IPS rubric、试点设计、评审流程

### 如果进入实现阶段
优先顺序建议：
1. 先做信息架构和文档骨架
2. 再做课程与实验室原型
3. 再做 IPS 表单与复盘闭环
4. 最后接入更强的个性化生成能力

## 与当前仓库的关系

如果后续在 iClaw 仓库内落地，优先考虑：
- `home-web/`：承载学习/品牌/入口页面
- `apps/`：承载桌面壳内学习产品表层
- OEM/control-plane 侧：承载内容元数据、学习配置、个性化规则

默认不要直接改 OpenClaw kernel，除非后续明确批准。
