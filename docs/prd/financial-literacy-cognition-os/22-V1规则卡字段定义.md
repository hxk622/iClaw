# 22. V1 规则卡字段定义

> 类型：Artifact Schema / V1  
> 目标：把“规则卡”从文案概念变成可结构化、可填写、可审计的产品对象

## 1. 为什么要单独定义字段

如果规则卡只是自由文本，后续很难：

- 做 Audit
- 做版本更新
- 做规则冲突检查
- 做 Lab 回写

因此 V1 规则卡应既保留文本自由度，又具备最小结构化能力。

## 2. V1 规则卡对象

建议对象名：

- `RuleCard`

建议包含：

- 基础信息
- 规则正文
- 适用场景
- 失效条件
- 风险边界
- 来源与版本

## 3. 字段定义

### 3.1 基础字段

#### `rule_id`

- 类型：string
- 用途：规则唯一 ID

#### `title`

- 类型：string
- 必填
- 用途：规则标题
- 示例：`我不因为短期涨幅而买入`

#### `rule_text`

- 类型：string
- 必填
- 用途：规则正文
- 示例：`当一个资产在 7–14 天内快速上涨时，我默认先观察，不立即追高。`

#### `status`

- 类型：enum
- 值：
  - `draft`
  - `active`
  - `revised`
  - `retired`

### 3.2 规则语义字段

#### `trigger_type`

- 类型：enum
- 用途：定义触发场景
- 建议值：
  - `price_surge`
  - `sharp_drawdown`
  - `hot_narrative`
  - `expert_signal`
  - `unclear_asset_role`
  - `other`

#### `if_condition`

- 类型：string
- 必填
- 用途：把规则写成 If-Then 的前半句
- 示例：`如果某主题资产在两周内大涨且我无法清楚解释其长期逻辑`

#### `then_action`

- 类型：string
- 必填
- 用途：规则动作
- 示例：`那么我不追高，只记录参与条件并进入观察`

#### `because_reason`

- 类型：string
- 推荐必填
- 用途：说明规则背后的认知理由
- 示例：`因为上涨不等于确定性，热度会放大我的错失焦虑`

### 3.3 风险与边界字段

#### `max_tolerated_drawdown`

- 类型：number | null
- 用途：若适用，记录该类决策可承受回撤阈值

#### `time_horizon`

- 类型：enum
- 建议值：
  - `days`
  - `weeks`
  - `months`
  - `years`

#### `invalidation_signal`

- 类型：string
- 推荐必填
- 用途：定义何时承认规则不适用或本次判断失效
- 示例：`如果我的买入理由无法脱离短期趋势独立成立，则该决策无效`

### 3.4 审计字段

#### `source_module`

- 类型：string
- 用途：这条规则来自哪个课程或实验
- 示例：
  - `module-1-finance-is-probability`
  - `lab-1-fomo-chase`

#### `source_type`

- 类型：enum
- 建议值：
  - `course`
  - `lab`
  - `audit`
  - `manual`

#### `revision_count`

- 类型：number
- 用途：记录这条规则被修改多少次

#### `last_revised_at`

- 类型：datetime | null

#### `conflict_flags`

- 类型：string[]
- 用途：记录被 Audit 标记过的冲突标签
- 示例：
  - `fomo_conflict`
  - `no_invalidation_signal`
  - `time_horizon_mismatch`

## 4. V1 最小必填字段

第一版为了降低门槛，建议只强制要求：

- `title`
- `rule_text`
- `if_condition`
- `then_action`
- `because_reason`
- `invalidation_signal`

如果用户不写 `invalidation_signal`，则不能判为“可审计规则”。

## 5. V1 默认规则卡模板

建议首版默认生成 5 张卡位：

1. 热点参与规则
2. 回撤应对规则
3. 外部观点采纳规则
4. 资产角色清晰规则
5. 不赚认知外的钱规则

## 6. 示例

### 示例 1：热点参与规则

- `title`：我不因为短期涨幅而买入
- `if_condition`：如果某主题资产在短期内快速上涨，且我无法独立说明其长期逻辑
- `then_action`：那么我不追高，只记录参与条件并继续观察
- `because_reason`：因为热度容易放大错失焦虑，而不是提升资产确定性
- `invalidation_signal`：如果后续能补齐资产角色、风险边界和失效条件，我再重新评估

### 示例 2：回撤应对规则

- `title`：我先看预案，再做动作
- `if_condition`：如果持有资产在短时间内出现明显回撤
- `then_action`：那么我先回看原始假设和预案，不做冲动卖出
- `because_reason`：因为波动不等于逻辑失效
- `invalidation_signal`：如果原始假设被证据否定，则进入减仓或退出评估

## 7. 与其他模块的关系

### 课程

课程负责让用户理解为什么需要规则。

### Lab

Lab 负责暴露用户现有规则是否失效。

### Audit

Audit 负责检查：

- 规则是否具体
- 规则是否可执行
- 行为是否违反规则

## 8. 产品实现建议

### UI 层

规则卡建议采用“半结构化表单”：

- 左边给句式模板
- 右边让用户填具体内容

避免完全空白文本框导致门槛过高。

### 数据层

V1 可以先存为 JSON 对象数组，不必过度抽象。

但字段名应尽量平台中性，以便未来迁移到其他认知学校。

## 9. V1 成功标准

规则卡系统成功，不是看用户写得多漂亮，而是看：

1. 是否至少生成 3 条可审计规则
2. 是否包含失效条件
3. 是否能在 Lab 后被修改
4. 是否能被 Audit 正确引用与对照
