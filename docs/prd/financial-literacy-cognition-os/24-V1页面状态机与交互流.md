# 24. V1 页面状态机与交互流

> 目标：把 V1 金融学校从“页面列表”推进到“可实现状态机”  
> 关联文档：
> - [18-核心页面低保真 Wireframe](./18-核心页面低保真Wireframe.md)
> - [21-V1 首个 Lab 完整脚本：热潮追高实验](./21-V1首个Lab完整脚本：热潮追高实验.md)
> - [23-V1 认知审计 Rubric](./23-V1认知审计Rubric.md)

## 1. 为什么需要状态机

如果只有页面 wireframe，没有状态机，研发会遇到三个问题：

- 不知道用户从哪里进入下一步
- 不知道什么条件下允许跳转
- 不知道什么时候必须强制进入 Audit / Rule Repair

V1 的关键不是页面多，而是：

> 课程、Lab、Artifact、Audit 必须按闭环顺序发生，而不是让用户随便逛。

## 2. V1 的顶层状态

建议 V1 用一个顶层学习状态机驱动：

```text
diagnose
  -> lesson
  -> rule-card-init
  -> lab
  -> audit
  -> rule-repair
  -> dashboard-next-step
```

其中：

- `diagnose`：初始短测与认知画像
- `lesson`：第一课
- `rule-card-init`：产出初始规则卡
- `lab`：首个压力实验
- `audit`：认知审计
- `rule-repair`：修改规则卡
- `dashboard-next-step`：回到首页，系统给出下一步

## 3. V1 全局路由建议

在产品层建议保留以下学习子页面：

- `learning/dashboard`
- `learning/map`
- `learning/lesson/:lessonId`
- `learning/lab/:labId`
- `learning/audit/:auditId`
- `learning/rule-card`
- `learning/review/:reviewId`

注意：

- `audit` 不建议隐藏在弹层里
- `rule-card` 不建议只作为 IPS 大页的一个角落
- 首个闭环中，`audit` 与 `rule-card` 都必须具有独立页面地位

## 4. 页面级状态机

## 4.1 Dashboard

### 页面状态

- `idle`：正常展示当前状态
- `has-required-next-step`：存在必须动作
- `blocked-by-audit`：用户先完成审计前，不允许进入下一非相关任务
- `ready-for-next-loop`：上一轮完成，等待下一步

### 主要输入

- 当前学习阶段
- 未完成课时
- 未完成 Lab
- 未完成 Audit
- 未完成 Rule Repair

### 主要 CTA 规则

- 若有未完成 `audit`，主 CTA = `进入认知审计`
- 若有未完成 `rule-repair`，主 CTA = `修改规则卡`
- 若 lesson 未完成，主 CTA = `继续课程`
- 若 lesson 完成且 lab 未开始，主 CTA = `进入实验室`

### 禁止逻辑

- 存在未完成审计时，不把“继续下一课”放在首位

---

## 4.2 Lesson Detail

### 页面状态

- `not-started`
- `in-progress`
- `knowledge-check-pending`
- `completed-awaiting-rule-init`

### 完成条件

本课只有同时满足以下条件才进入 `completed-awaiting-rule-init`：

1. 用户完成本课阅读/讲解
2. 完成最小思维检查
3. 写出一句“金融不是预测，而是概率管理”的个人重述

### 出口动作

课后主 CTA 不应直接是“返回首页”，而应是：

- `生成第一版规则卡`

---

## 4.3 Rule Card Init / Rule Card Workspace

### 页面状态

- `empty`
- `drafting`
- `minimum-viable-complete`
- `submitted-for-lab`
- `revision-required`

### V1 最小完成标准

进入 `minimum-viable-complete` 必须满足：

- 至少 2 条规则卡草稿
- 每条都包含：
  - `if_condition`
  - `then_action`
  - `because_reason`
  - `invalidation_signal`

### 主 CTA 规则

- 未满足最小完成标准：`继续补全规则卡`
- 满足后：`进入实验室`

### 禁止逻辑

- 没有最小规则卡时，不允许直接进入首个 Lab

这样 Audit 才有东西可对照。

---

## 4.4 Lab

### 页面状态

- `briefing`
- `decision-open`
- `decision-submitted`
- `timeline-reveal`
- `reflection-required`
- `completed-awaiting-audit`

### 状态流

```text
briefing
  -> decision-open
  -> decision-submitted
  -> timeline-reveal
  -> reflection-required
  -> completed-awaiting-audit
```

### 每个状态的关键要求

#### `briefing`
- 展示场景背景
- 显示本次实验要测试什么偏差
- 不提前给答案

#### `decision-open`
- 允许 30 秒首轮决策
- 必须填写决策依据

#### `decision-submitted`
- 锁定首轮选择
- 不允许直接改答案

#### `timeline-reveal`
- 系统推进 T+1 / T+7 / T+21
- 用户只能响应，不可回写首轮理由

#### `reflection-required`
- 强制回答复盘问题

#### `completed-awaiting-audit`
- Lab 已结束
- 唯一主 CTA = `进入认知审计`

---

## 4.5 Audit

### 页面状态

- `assembling`
- `showing-conflict`
- `awaiting-user-position`
- `repair-required`
- `completed`

### 状态说明

#### `assembling`
- 聚合输入：
  - 决策记录
  - 规则卡
  - 复盘文本
  - 标签与评分

#### `showing-conflict`
- 系统展示：
  - 你做了什么
  - 你基于什么做的
  - 你和哪条规则冲突

#### `awaiting-user-position`
- 强制用户表态：
  - 我承认这是规则失守
  - 我要修改规则
  - 我认为规则本身需要升级

#### `repair-required`
- 审计页不直接结束
- 主 CTA = `去修改规则卡`

#### `completed`
- 只有在用户完成 repair 后才达成

### 核心约束

Audit 不是结果页，而是一个强制中间页。

---

## 4.6 Rule Repair

### 页面状态

- `pending`
- `editing`
- `diff-preview`
- `saved`

### 关键动作

- 显示旧规则
- 显示本次冲突点
- 让用户重写规则
- 显示前后 diff

### 完成条件

至少有一个明确变化：

- 新增规则
- 修改 `if_condition`
- 修改 `then_action`
- 修改 `invalidation_signal`

如果只是点“知道了”不修改，不算 repair 完成。

---

## 4.7 Review

### 页面状态

- `summary`
- `knowledge-gap-linked`
- `next-step-generated`

### 页面职责

- 总结这一轮闭环发生了什么
- 指向对应知识缺口
- 生成下一步 CTA

它是本轮收尾页，不是认知核心页。

## 5. 全局强制流转规则

V1 必须有以下强制顺序：

```text
Lesson Completed
  -> Rule Card Init
  -> Lab
  -> Audit
  -> Rule Repair
  -> Dashboard
```

不允许：

- 课后直接跳过规则卡
- Lab 后直接回首页
- Audit 后不修规则

## 6. V1 最小交互闭环

如果只做最小版，建议保留以下 8 个关键交互动作：

1. Lesson 完成后的个人重述
2. 初始规则卡填写
3. Lab 30 秒首答
4. Lab 强制写依据
5. 时间推进
6. Audit 展示规则冲突
7. Rule Repair 前后 diff
8. Dashboard 给出下一步

少于这 8 个动作，闭环会断。

## 7. 与当前桌面壳的结合要求

结合现有实现现状：

- 当前 `apps/desktop/src/app/lib/chat-navigation-resolution.ts` 只显式识别 `/chat`
- 当前桌面壳已有 `primaryView` 概念与 `menuUiConfig`
- 因此 V1 第一版更适合：
  - 先接入一个新的 `primaryView`
  - 再逐步扩展 URL / 深链解析

建议：

- 第一版优先走 shell 菜单选择 + 本地持久化
- 第二版再补 `learning/*` 的深链

## 8. 这份状态机达到“可实现”标准的定义

如果研发能够仅凭本页回答以下问题，就说明规格够用了：

1. 哪些页面是 V1 必需页面？
2. 页面何时可进入、何时必须阻塞？
3. Audit 为什么不是普通结果页？
4. Rule Repair 如何判定完成？
5. Dashboard 的主 CTA 如何由状态推导？

如果还回答不出，说明规格仍不够。
