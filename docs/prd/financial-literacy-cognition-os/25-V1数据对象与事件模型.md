# 25. V1 数据对象与事件模型

> 目标：把 V1 认知闭环翻译成研发可实现的数据对象与事件流  
> 关联文档：
> - [22-V1 规则卡字段定义](./22-V1规则卡字段定义.md)
> - [23-V1 认知审计 Rubric](./23-V1认知审计Rubric.md)
> - [24-V1 页面状态机与交互流](./24-V1页面状态机与交互流.md)

## 1. 设计原则

V1 数据层要满足四个要求：

1. 能支撑首个金融闭环完整跑通
2. 能记录前后版本差异
3. 能让 Audit 读到“规则—行为冲突”
4. 字段名尽量平台中性，不把底层绑死在金融术语上

因此建议：

- 垂类文案可以金融化
- 底层对象命名尽量认知 OS 化

## 2. V1 必需对象

V1 最少要有 8 个对象：

1. `LearningProfile`
2. `LessonProgress`
3. `KnowledgeNote`
4. `RuleCard`
5. `LabScenario`
6. `LabRun`
7. `AuditRecord`
8. `ReviewTask`

## 3. 对象定义

## 3.1 LearningProfile

用户在当前学校中的整体状态。

```ts
type LearningProfile = {
  profile_id: string;
  user_id: string;
  school_slug: "financial-school";
  current_phase: "diagnose" | "lesson" | "lab" | "audit" | "repair";
  current_module_id: string | null;
  current_lesson_id: string | null;
  active_lab_run_id: string | null;
  pending_audit_id: string | null;
  rule_card_completion_ratio: number;
  last_detected_blindspots: string[];
  created_at: string;
  updated_at: string;
};
```

用途：

- Dashboard
- 主 CTA 推导
- 学习态持久化

---

## 3.2 LessonProgress

记录课时完成度与输出物。

```ts
type LessonProgress = {
  lesson_progress_id: string;
  user_id: string;
  lesson_id: string;
  status: "not_started" | "in_progress" | "completed";
  knowledge_check_score: number | null;
  personal_restated_thesis: string | null;
  identified_old_bias: string | null;
  created_rule_card_ids: string[];
  completed_at: string | null;
  updated_at: string;
};
```

关键点：

- `personal_restated_thesis` 是第一课能否真正完成的证据

---

## 3.3 KnowledgeNote

V1 的本地 Wiki 轻量对象。

```ts
type KnowledgeNote = {
  note_id: string;
  user_id: string;
  note_type: "concept" | "bias" | "lesson_output" | "lab_reflection";
  title: string;
  body: string;
  tags: string[];
  source_ref: {
    source_type: "lesson" | "lab" | "audit" | "manual";
    source_id: string;
  };
  created_at: string;
  updated_at: string;
};
```

V1 不需要完整 wiki engine，但至少要有这个统一落点。

---

## 3.4 RuleCard

`RuleCard` 详细字段已在 [22](./22-V1规则卡字段定义.md) 定义。  
本页只强调它在闭环中的角色：

- Lesson 输出
- Lab 对照物
- Audit 冲突对象
- Repair 修改目标

除当前版本外，建议保留快照：

```ts
type RuleCardSnapshot = {
  snapshot_id: string;
  rule_id: string;
  version: number;
  title: string;
  rule_text: string;
  if_condition: string;
  then_action: string;
  because_reason: string;
  invalidation_signal: string;
  source_trigger: "lesson_init" | "lab_repair" | "manual_edit";
  created_at: string;
};
```

---

## 3.5 LabScenario

预定义实验脚本。

```ts
type LabScenario = {
  lab_id: string;
  school_slug: "financial-school";
  slug: "fomo-chase";
  title: string;
  target_biases: string[];
  required_rule_topics: string[];
  timeline_steps: Array<{
    step_id: string;
    label: string;
    narrative: string;
  }>;
  reflection_questions: string[];
  created_at: string;
  updated_at: string;
};
```

关键点：

- `required_rule_topics` 用于判断进入 Lab 前规则卡是否足够

---

## 3.6 LabRun

一次用户真实实验过程。

```ts
type LabRun = {
  lab_run_id: string;
  user_id: string;
  lab_id: string;
  status:
    | "briefing"
    | "decision_open"
    | "decision_submitted"
    | "timeline_reveal"
    | "reflection_required"
    | "completed";
  first_decision: "buy" | "probe" | "observe" | "skip" | null;
  first_decision_deadline_sec: number | null;
  decision_reason_text: string | null;
  key_variable_text: string | null;
  possible_failure_text: string | null;
  downside_plan_text: string | null;
  timeline_position: "t0" | "t1" | "t7" | "t21";
  reflection_answers: Array<{
    question: string;
    answer: string;
  }>;
  linked_rule_card_ids: string[];
  created_at: string;
  updated_at: string;
  completed_at: string | null;
};
```

这是 Audit 的核心输入对象。

---

## 3.7 AuditRecord

一次认知审计结果。

```ts
type AuditRecord = {
  audit_id: string;
  user_id: string;
  lab_run_id: string;
  status: "assembling" | "awaiting_user_position" | "repair_required" | "completed";
  labels: string[];
  score_breakdown: {
    reasoning_clarity: number;
    risk_invalidation_awareness: number;
    rule_consistency: number;
    repair_willingness: number | null;
  };
  conflict_items: Array<{
    rule_id: string;
    conflict_type: string;
    evidence_excerpt: string;
  }>;
  coach_feedback: string[];
  user_position:
    | "admit_rule_break"
    | "repair_rule"
    | "upgrade_rule"
    | null;
  repair_required_rule_ids: string[];
  created_at: string;
  updated_at: string;
  completed_at: string | null;
};
```

关键点：

- `conflict_items` 必须能点名具体规则
- `user_position` 是闭环里不可缺失的用户表态

---

## 3.8 ReviewTask

审计后的下一步任务。

```ts
type ReviewTask = {
  review_task_id: string;
  user_id: string;
  source_type: "audit";
  source_id: string;
  task_type: "repair_rule" | "revisit_lesson" | "rerun_lab";
  title: string;
  description: string;
  target_ref: {
    ref_type: "rule_card" | "lesson" | "lab";
    ref_id: string;
  };
  status: "open" | "done";
  created_at: string;
  updated_at: string;
};
```

用途：

- Dashboard 下一步
- Review 页 CTA

## 4. 对象关系图

```text
LearningProfile
  -> LessonProgress
  -> RuleCard
  -> LabRun
  -> AuditRecord
  -> ReviewTask

LessonProgress
  -> creates RuleCard

LabScenario
  -> instantiates LabRun

LabRun
  -> feeds AuditRecord

AuditRecord
  -> requires RuleCard repair
  -> creates ReviewTask
```

## 5. V1 关键事件流

建议 V1 同时记录结构化对象与事件流。

### 关键事件列表

```ts
type LearningEvent =
  | { type: "lesson_started"; lesson_id: string; at: string }
  | { type: "lesson_completed"; lesson_id: string; at: string }
  | { type: "rule_card_created"; rule_id: string; at: string }
  | { type: "lab_started"; lab_run_id: string; lab_id: string; at: string }
  | { type: "lab_first_decision_submitted"; lab_run_id: string; decision: string; at: string }
  | { type: "lab_reflection_completed"; lab_run_id: string; at: string }
  | { type: "audit_generated"; audit_id: string; at: string }
  | { type: "audit_position_selected"; audit_id: string; position: string; at: string }
  | { type: "rule_card_revised"; rule_id: string; version: number; at: string }
  | { type: "loop_completed"; audit_id: string; at: string };
```

事件流的意义：

- 支撑学习日志
- 支撑未来 wiki 编译
- 支撑后续复盘与指标

## 6. V1 关键派生状态

很多 UI 状态不必单独存，可以由对象推导：

### `hasPendingAudit`

条件：

- 存在 `AuditRecord.status !== "completed"`

### `hasRequiredRuleRepair`

条件：

- `AuditRecord.status === "repair_required"`

### `canStartFirstLab`

条件：

- 第一课完成
- 规则卡满足最小完整度

### `nextPrimaryCTA`

来源：

- `LearningProfile`
- `LessonProgress`
- `LabRun`
- `AuditRecord`
- `ReviewTask`

## 7. 第一版存储建议

### 客户端本地

V1 可以先采用：

- JSON 持久化
- 本地 cache / local db
- 规则卡版本快照

### 后续同步

后面再考虑：

- 跨端同步
- 后台分析
- OEM 配置差异化

当前不要为了平台化过度设计远程 schema。

## 8. 这份数据模型达到“够实现”的标准

如果研发能据此回答以下问题，就算达标：

1. Lesson 完成后，什么对象证明用户真的完成了第一课？
2. Lab 的首轮选择和理由存在哪里？
3. Audit 如何知道用户违反了哪条规则？
4. Rule Repair 如何保留前后版本差异？
5. Dashboard 的下一步 CTA 由哪些对象推导？

若这些问题仍含糊，就说明模型还不够。
