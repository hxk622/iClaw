# 09. 验收标准与验证方案

## 核心成功标准

这份文档描述的是金融垂类的成功标准，但它现在必须服从平台层更高一级的判断：

> 同一套 Wiki / Lab / Artifact / Audit 底层结构，能否完整支撑 Financial School 的真实闭环。

在金融垂类内，最关键的结果指标仍然是：

> 在首批 100 个目标用户中，至少 30 人能独立写出一份逻辑闭环、可审计、不自相矛盾的 IPS。

## 验收标准

1. 产品对外定位必须明确为教育/认知系统
2. 产品界面和流程中不能出现荐股、代下单、收益承诺
3. 用户能在 90 天内走完核心学习路径
4. 每个课程模块必须包含：
   - 原理讲解
   - 图解
   - 思维检查
   - 场景应用
5. 认知实验室至少包含 3 个高信号场景
6. IPS 必须要求用户写下风险边界、资产角色、时间周期和 If-Then 规则
7. 必须存在一套 IPS 评审 rubric

## 需要采集的证据

- 初始认知诊断结果
- 课程完成情况
- 小测与推理题表现
- 模拟实验中的决策记录
- IPS 初稿与修改历史
- IPS rubric 评分结果
- 常见失败模式归因

## 验证流程

1. 先定义 IPS 评分 rubric
2. 招募 100 个目标用户进入试点
3. 跑完整个 90 天闭环
4. 对最终 IPS 做盲审
5. 统计通过率
6. 对失败样本进行原因归纳，反哺课程与实验室设计

## 次级指标

这些可以看，但不是第一判断标准：
- 完课率
- 30/90 天留存
- IPS 重写频次
- 某类情境错误率是否下降


## 与平台层的关系

- 平台层负责定义闭环协议：Wiki / Lab / Artifact / Audit
- 金融层负责定义 IPS、金融课程、金融实验脚本和金融专属验证
- 因此 IPS 通过率是金融层指标，不等于平台层整体成功标准
