# 垂类金融 AI 桌面端测试计划 v2.0

## 1. 项目背景

当前产品是基于 OpenClaw 封装的金融垂类 AI 桌面端软件，参考 `EasyClaw` 的产品形态，目标是服务 C 端用户，降低安装和使用门槛，同时提供金融投研、问答、资料解析、辅助分析等能力。

测试工作的核心，不只是验证“能不能用”，而是验证：

1. 桌面端是否稳定、易安装、易恢复。
2. 金融 AI 输出是否可信、可控、可追溯。
3. RAG/知识问答是否能正确处理金融文档。
4. 产品是否满足金融场景下对准确性、合规性、数据隔离的要求。

## 2. 测试目标

1. 验证桌面端安装、启动、升级、卸载全链路稳定可用。
2. 验证 OpenClaw 封装层与底层模型/服务交互稳定，无明显异常。
3. 验证金融知识问答、文档解析、RAG 检索、上下文管理的正确性。
4. 验证在弱网、断网、长时间运行、大文件输入下的鲁棒性。
5. 验证金融场景中的高风险问题：
   - 幻觉
   - 数值错误
   - 时点错误
   - 不当投资建议
   - 敏感数据泄露
6. 建立可持续复用的自动化与评测基线，支撑后续版本迭代。

## 3. 测试范围

### 3.1 In Scope

1. 桌面端基础能力
   - 安装、启动、登录、退出、升级、卸载
   - 本地配置、本地缓存、本地历史记录
   - 异常崩溃恢复

2. 核心业务功能
   - 对话问答
   - 金融垂类 Agent 切换
   - 历史会话管理
   - 文件上传、解析、问答
   - 金融文档 RAG 检索与引用回答

3. AI 能力专项
   - 金融知识问答准确性
   - 多轮对话上下文保持
   - 超长文本理解
   - 表格/财报/研报解析
   - 引用溯源能力

4. 非功能专项
   - 性能
   - 稳定性
   - 兼容性
   - 弱网/断网恢复
   - 数据隔离与安全

### 3.2 Out of Scope

1. 基础大模型本身的训练质量评估。
2. 底层开源内核算法改造验证。
3. 非当前产品形态之外的移动端或网页端专项。

## 4. 风险分析

该项目的主要风险不在传统 UI，而在“金融 + AI”组合带来的高风险输出问题。

### 4.1 业务风险

1. 回答看似合理但事实错误。
2. 财务指标、数值、涨跌幅、日期引用错误。
3. 把信息检索型问题回答成投资建议型结论。
4. 上传文档后回答不基于文档，出现幻觉。
5. 多轮对话后上下文漂移。

### 4.2 产品风险

1. 安装复杂、依赖缺失、权限异常。
2. 长时间运行后卡顿、白屏、崩溃。
3. 大文件解析耗时过长或失败。
4. 断网/模型超时后体验差，状态不可恢复。

### 4.3 合规与安全风险

1. 输出带有诱导性荐股表述。
2. 缺失必要风险提示。
3. 会话间数据串扰。
4. 用户上传资料、持仓信息、个人信息泄露。

## 5. 测试策略

### 5.1 测试方法

采用“手工测试 + 自动化回归 + AI 评测 + 风险专项”的组合策略。

1. 手工测试
   - 用于新功能探索、体验验证、异常场景挖掘
2. 自动化测试
   - 用于主流程冒烟和核心回归
3. AI 评测
   - 用于评估回答质量、忠实度、相关性、数值准确性
4. 风险专项测试
   - 用于验证金融高风险场景和合规边界

### 5.2 测试分层

1. UI/端到端层
   - 安装、启动、登录、问答、上传文档、查看历史
2. 服务交互层
   - 请求/响应、超时、重试、异常码处理
3. RAG 链路层
   - 文档解析、分块、向量化、检索、生成
4. 模型输出层
   - 忠实度、相关性、数值准确性、引用准确性
5. 稳定性与安全层
   - 长稳、资源占用、数据隔离、异常恢复

## 6. 重点测试内容

### 6.1 桌面端基础测试

1. 首次安装是否顺畅。
2. Windows/macOS 是否均可正常运行。
3. 启动速度是否可接受。
4. 升级安装是否保留用户数据。
5. 卸载后残留是否符合预期。
6. 崩溃后重启是否能恢复会话或至少不损坏数据。

### 6.2 核心对话功能测试

1. 普通问答是否可正常返回。
2. 连续多轮对话后语义是否连续。
3. 切换 Agent 后上下文是否正确隔离。
4. 历史会话查看、删除、恢复是否正常。
5. 中途取消生成、重试生成是否正常。

### 6.3 金融 AI 专项测试

1. 金融基础知识问答
   - 财报指标
   - 宏观术语
   - 行业术语
2. 金融研报理解
   - 总结结论
   - 提取观点
   - 提取风险提示
3. 数值抽取与比较
   - 营收、利润、毛利率、同比、环比
4. 时点一致性
   - 区分历史数据、预测数据、最新披露日期
5. 结论边界控制
   - 不应把资讯摘要输出为明确投资建议

### 6.4 RAG 与文档解析测试

1. 支持格式
   - PDF
   - Excel
   - CSV
2. 文档类型
   - 年报
   - 季报
   - 券商研报
   - 数据表
3. 文档异常
   - 扫描版 PDF
   - 超长 PDF
   - 跨页表格
   - 图文混排
   - 空表格/脏数据/错列
4. 验证点
   - 是否解析成功
   - 是否正确分块
   - 检索结果是否命中关键证据
   - 回答是否引用到正确内容
   - 是否存在“文档未提及却编造”的问题

### 6.5 异常场景测试

1. 网络抖动。
2. 模型接口超时。
3. 限流。
4. 本地磁盘空间不足。
5. 上传中断。
6. 大文件处理失败。
7. 应用异常退出。
8. 服务不可用后的恢复路径。

### 6.6 安全与隔离测试

1. 用户 A 上传的文件不会被用户 B 问答命中。
2. 不同会话的上下文不会串扰。
3. 敏感信息是否被错误展示在日志/UI 中。
4. 本地缓存是否可控。
5. 删除会话后数据是否符合预期处理。

### 6.7 合规与高风险表达测试

重点验证模型不会轻易输出：

1. 保证收益。
2. 明确荐股。
3. 确定性涨跌判断。
4. 替代持牌投顾的建议。
5. 对不完整信息给出过度确定结论。

## 7. 测试数据设计

### 7.1 数据集分层

建议至少维护 4 类数据集：

1. 基准集
   - 标准财报、标准研报、结构清晰数据表
   - 用于日常回归
2. 对抗集
   - 扫描件、超长文档、格式错乱表格、图片型 PDF
   - 用于鲁棒性测试
3. 风险集
   - 诱导荐股、收益承诺、内幕信息、账户隐私相关问题
   - 用于合规测试
4. 缺陷回归集
   - 来自线上或历史 bug 的固定问题集
   - 用于防回归

### 7.2 样本维度

每条测试样本建议包含：

1. 问题
2. 输入上下文/文档
3. 标准答案或关键点
4. 风险等级
5. 预期行为
6. 是否允许开放式回答
7. 是否要求引用证据

## 8. 质量评估指标

### 8.1 功能指标

1. P0 用例通过率。
2. P1 用例通过率。
3. 安装成功率。
4. 崩溃率。
5. 断网恢复成功率。

### 8.2 AI 质量指标

1. 忠实度 Faithfulness。
2. 相关性 Relevance。
3. 数值准确率。
4. 引用准确率。
5. 幻觉率。
6. 上下文丢失率。

### 8.3 性能指标

1. 首次启动时间。
2. 问答首 token 响应时间。
3. 长文档解析耗时。
4. 内存峰值。
5. 24 小时运行内存增长趋势。

### 8.4 合规指标

1. 高风险回答命中率。
2. 敏感表达拦截率。
3. 越权建议发生数。
4. 数据串扰发生数。

## 9. 测试阶段安排

### 阶段一：冒烟与基础能力

目标：

1. 跑通安装、启动、登录、基础问答。
2. 建立 P0 冒烟用例。
3. 初步覆盖文档上传与基础解析。

出口标准：

1. P0 主流程全部可跑通。
2. 无阻塞级崩溃问题。
3. 冒烟脚本可稳定执行。

### 阶段二：金融问答与 RAG 专项

目标：

1. 完成金融问答正确性测试。
2. 建立文档解析与 RAG 数据集。
3. 输出第一版 AI 评测结果。

出口标准：

1. RAG 主流程稳定。
2. 高风险幻觉问题被识别。
3. 关键金融文档类型均已覆盖。

### 阶段三：异常、稳定性、性能

目标：

1. 覆盖弱网、超时、断连、失败恢复。
2. 完成长稳测试。
3. 输出性能基线。

出口标准：

1. 长稳运行无重大异常。
2. 弱网与恢复路径清晰。
3. 性能指标达到版本要求。

### 阶段四：回归与发布验收

目标：

1. 执行全量回归。
2. 验证 bug 修复有效。
3. 形成上线测试结论。

出口标准：

1. P0/P1 缺陷收敛。
2. 无未评估高风险合规问题。
3. 测试报告可支持发布决策。

## 10. 缺陷分级建议

1. P0
   - 无法安装/无法启动
   - 核心对话不可用
   - 文档上传后系统崩溃
   - 明显违规金融建议
   - 严重数据泄露/串话
2. P1
   - 关键文档解析失败
   - 多轮上下文严重错乱
   - 数值结论明显错误
   - 高频场景下稳定复现的功能异常
3. P2
   - 体验问题
   - 偶发错误
   - 次要展示异常
4. P3
   - 文案、样式、小概率边缘问题

## 11. 自动化建议

你的项目很适合尽早建立自动化，但不要一开始就追求“大而全”。

建议优先顺序：

1. 冒烟自动化
   - 启动应用
   - 登录
   - 发起问答
   - 上传一个标准文档
   - 验证有返回结果
2. 核心回归自动化
   - 历史会话
   - Agent 切换
   - 文件上传与解析
   - 断网重试
3. AI 评测自动化
   - 固定问题集批量跑分
   - 输出忠实度、相关性、数值正确率
4. 缺陷回归自动化
   - 线上问题沉淀成固定 case

## 12. 测试建议

结合当前岗位职责，以下建议最重要：

1. 不要把金融 AI 测试只做成功能测试。
真正有价值的是“输出质量测试”和“高风险场景测试”。

2. 先建小而稳的基准集。
不用一开始就做几千条样本，先做 50 到 100 条高质量金融基准题，比大量低质量样本更有效。

3. 一定要单独盯“数值错误”。
金融用户最不能接受的不是文风差，而是数字错、日期错、涨跌幅错。

4. 把“合规边界”拉进测试主流程。
不要等法务或产品最后看，测试阶段就要提前设计诱导性问题和危险表达样本。

5. RAG 必须分层看。
不要只看最后答案。要区分是“解析错了”“没召回到”还是“模型编了”。

6. 建立可复现机制。
每个 AI 缺陷尽量沉淀为固定输入、固定文档、固定预期，不然很难回归。

7. 测试报告要能支持发布决策。
最终报告不要只写“测了什么”，要写清楚：
   - 当前是否可以上线
   - 剩余风险是什么
   - 风险是否可接受
   - 哪些场景不建议开放给用户
