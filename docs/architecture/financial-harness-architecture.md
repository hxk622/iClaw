# Financial Harness Architecture

更新时间：2026-04-18

## 1. 目标

定义 `financial harness` 在 `iClaw / 理财客` 中的职责和边界，明确它不是一个可以自由扩权、自由执行、自由给建议的 agent，而是一套把模型能力包裹在金融约束、渠道约束、数据约束和审计约束之内的运行架构。

这份文档回答的不是“模型能说什么”，而是：

- 什么能力允许进入金融场景
- 什么问题允许被回答到什么深度
- 什么输出必须降级、提示或拦截
- 什么渠道可以展示什么级别的金融内容
- 为什么最终行为由架构裁决，而不是由 agent 自行决定

## 2. 一句话定义

`financial harness` 不是“会金融的 agent”。

`financial harness` 是一层 **受约束的金融运行框架**：它把模型、工具、数据源、页面渠道和 OEM 策略装进一组可验证的护栏里，使系统输出始终停留在“研究参考与信息整理”边界内，而不会滑向“无约束的个性化投顾或交易代理”。

## 3. 它不是什么

`financial harness` 明确不等于以下东西：

- 不是自动投顾
- 不是持牌适当性判断器
- 不是交易执行代理
- 不是“模型想怎么答就怎么答”的聊天壳
- 不是靠 prompt 自觉守规矩的软约束系统

这意味着它的核心不是“更聪明的回答”，而是“更可控的回答”。

## 4. 核心架构命题

在金融场景里，真正应该被信任的不是 agent 的自觉，而是架构的约束能力。

因此 `financial harness` 采用以下基本命题：

1. 能力先准入，再使用
2. 输入先分类，再进入生成
3. 输出先判断风险，再决定展示方式
4. 渠道先看允许范围，再决定是否分发
5. 全链路必须留痕，事后可回放

换句话说，agent 只是参与推理的一环，不是规则的拥有者，更不是最终裁决者。

## 5. 总体分层

```text
Platform Catalog / OEM Binding
  -> Capability Admission
     -> Input Classification
        -> Model / Tool Orchestration
           -> Output Classification
              -> Degrade / Disclaimer / Block
                 -> Surface-specific Rendering
                    -> Audit Trail
```

每一层都在缩小 agent 的自由度：

- catalog 决定它能用什么
- classification 决定它能回答到哪一层
- wrapper 决定它最终能不能展示
- audit 决定它不能“做了却不留痕”

## 6. 金融领域的约束边界

## 6.1 能力边界：不是所有能力都能进金融场景

`financial harness` 的第一条边界不是回答内容，而是能力准入。

只有进入平台 catalog 并带有金融合规元数据的能力，才允许被金融场景装配。能力至少要回答这些问题：

- 它是 `data_only`、`research_only`、`investment_view`，还是更高风险档位
- 它是否依赖市场数据、外部研究、专家观点或高敏个人信息
- 它是否存在执行导向
- 它是否要求来源、时点、延时披露

因此 agent 没有“临场给自己加能力”的空间：

- 不能私自启用未准入的 skill / MCP / model
- 不能绕过 OEM binding 使用不属于当前金融产品的能力
- 不能把通用能力临时伪装成金融合规能力

## 6.2 建议边界：允许研究，不允许越界成投顾

`financial harness` 允许：

- 市场信息整理
- 金融概念解释
- 公司、基金、行业、宏观的研究分析
- 概率、情景、风险因子的结构化讨论

`financial harness` 不允许默认输出：

- 明确买卖指令
- 仓位建议
- 收益承诺
- 保本保收益表达
- 将研究观点包装成确定性结论

这条边界的本质是：

- 可以帮助用户理解决策
- 不能替用户完成受监管的投资建议动作

## 6.3 执行边界：只能分析，不能代替执行

在架构上，`financial harness` 必须和执行链路断开。

默认禁止：

- 下单
- 申赎
- 调仓执行
- 开户或资金划转
- 向券商、基金销售或交易系统发送可执行指令

即使用户提出“帮我直接买”“按这个比例调仓”，系统也只能：

- 拒绝
- 降级成研究框架
- 引导到受监管或人工确认链路

这说明它不是一个 end-to-end agent，而是一个被切断执行权的研究架构。

## 6.4 个性化边界：不能把用户画像直接变成适当性结论

金融系统最危险的滑坡，不是一般性的市场分析，而是把用户画像直接映射成“适合你”的建议。

因此 `financial harness` 默认禁止以下行为：

- 根据资产规模、风险偏好、年龄、期限直接输出个性化推荐
- 以“你应该买什么”替代适当性流程
- 用免责声明掩盖个性化建议事实

若用户输入触发：

- `personalized_request`
- `advice_request`
- `execution_request`

系统必须触发额外治理动作，而不是继续当作普通问答：

- 补充风险框架
- 降级成通用研究表达
- 拒绝回答到执行层
- 留下分类与裁决审计

## 6.5 数据边界：无来源、无时点、无延时说明的数据不能被直接消费

金融内容的风险不只在“说了什么”，也在“依据了什么数据”。

因此 `financial harness` 要求金融数据至少具备以下约束：

- 来源可识别
- 时间点可识别
- 延时属性可识别
- 原始数据与派生观点可区分

默认不应出现：

- 无来源行情
- 无时间戳观点
- 把旧数据包装成实时结论
- 把页面展示字段误当成事实真值

这条边界保证系统不是“想到哪说到哪”，而是“基于明确时点和来源说到哪”。

## 6.6 渠道边界：聊天能说的，不一定能推送

`financial harness` 不把所有输出视为同一风险面。

不同渠道必须分开治理：

- 聊天主回答
- 定时任务结果
- 通知摘要
- 首页卡片
- 专家卡片
- HTML 报告

例如：

- 聊天里允许展开风险前提
- 通知里不允许直接下强观点结论
- 报告必须展示生成时间、数据时间和免责声明
- 卡片内容必须比长文更弱、更保守

因此 agent 无法把一次高风险输出无损复制到所有分发面。渠道策略高于单次生成结果。

## 6.7 自主性边界：agent 没有自我扩权和自我改规权限

`financial harness` 里的 agent 只拥有受限的推理权限，不拥有制度权限。

它不能自行决定：

- 什么算合规
- 什么能力可用
- 什么渠道可发
- 什么级别必须免责声明
- 什么情况下可以跳过降级
- 什么日志可以不记

这些权力属于：

- platform catalog
- OEM binding
- compliance policy registry
- wrapper resolver
- audit store

因此这里的 agent 更像“被框在制度里的推理引擎”，而不是“拥有主权的自治体”。

## 6.8 审计边界：所有关键裁决都必须留痕

只要系统能：

- 分类
- 降级
- 拦截
- 展示免责声明
- 进入通知或报告

就必须能回答“为什么”。

所以 `financial harness` 至少要记录：

- 使用了哪些能力
- 命中了哪些规则
- 输入属于哪类金融请求
- 输出属于哪类风险档位
- 是否被降级
- 是否被拦截
- 是否展示免责声明
- 最终进入了哪个渠道

能被回放，才叫 harness；不能被回放，就仍然是黑箱 agent。

## 7. 运行时裁决原则

`financial harness` 的关键不是单点 prompt，而是运行时裁决链。

推荐采用以下顺序：

1. 能力准入检查
2. 输入分类
3. 生成与工具调用
4. 输出分类
5. wrapper 最终裁决
6. 渠道渲染或拦截
7. 审计落库

其中最关键的一点是：

**最终裁决权不在模型，不在单个 skill，也不在 plugin，而在 wrapper / policy resolver。**

这意味着即使模型生成了高风险内容，系统仍然可以：

- 降级
- 改写展示层
- 加强免责声明
- 阻止进入通知
- 直接拦截

## 8. 为什么它不是“无拘无束的 agent”

可以用一个简单判断标准来区分：

如果一个系统可以自己选工具、自己定规则、自己决定是否执行、自己决定是否留痕，那它是无拘无束的 agent。

`financial harness` 不是，因为它满足相反的 6 个条件：

1. 它不能自己扩能力
2. 它不能自己抬高建议等级
3. 它不能自己接执行链路
4. 它不能自己跳过免责声明和降级
5. 它不能自己决定所有渠道都可复用
6. 它不能自己擦除审计痕迹

因此它更准确的定义是：

一个带有金融制度护栏的 agent runtime，而不是一个自由行动的金融 agent。

## 9. 设计结论

如果要让用户、产品、法务和工程都相信这个系统“懂规矩”，光有免责声明不够，必须让约束写进架构本身。

`financial harness` 的价值就在这里：

- 它把“研究参考”与“执行建议”切开
- 它把“模型会说”与“系统允许展示”切开
- 它把“有能力”与“有权限”切开
- 它把“生成结果”与“渠道分发”切开

这些切分一旦存在，系统就不再是一个无拘无束的 agent，而是一个能被约束、能被解释、能被审计的金融运行架构。
