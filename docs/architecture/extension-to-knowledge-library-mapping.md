# iClaw 浏览器插件 -> iClaw 知识库 映射设计

## 目标

明确 iClaw 浏览器插件在 iClaw 体系中的职责边界，以及插件产生的数据如何进入知识库。

结论：

> iClaw 浏览器插件的定位是浏览器侧 Raw ingest runtime，不是 iClaw 内另一套独立知识产品。

## 1. 职责边界

### 插件负责

- 页面上下文采集
- 滑选摘录
- 视频时间戳摘录
- 本地去重与本地暂存
- 同步素材到 iClaw

### 知识库负责

- Raw 列表与详情
- Graph 编译与展示
- Output 生成与展示
- 第三栏 Chat 协作

## 2. 核心映射

### 页面级 capture

插件对象：`source`

映射到 iClaw：
- `RawMaterial(kind='source')`

用途：
- 在知识库 Raw tab 中展示页面来源对象

### 选区级 capture

插件对象：`snippet`

映射到 iClaw：
- `RawMaterial(kind='snippet')`

用途：
- 在知识库 Raw tab 中展示摘录对象
- 可进一步编译进图谱

### pick workflow

插件动作：`pick`

映射到 iClaw：
- 素材层工作流，不是独立产品页

用途：
- 用户在浏览器中临时挑选片段
- 进入客户端后仍然表现为 Raw 对象

## 3. 认证边界

### 重要约束

浏览器插件默认只能自动继承浏览器自身的登录态，不能直接继承 Tauri 桌面端安全存储中的 token。

因此正式产品路径中：

- 插件自动登录以桌面授权桥为主
- 不直接继承桌面主 token
- 浏览器登录只保留给开发 / 自测场景

## 4. 本地优先原则

插件必须保持 local-first：

- 先本地保存 source/snippet
- 再异步同步到 iClaw API

原因：

- 浏览器摘录动作必须立即反馈
- 不能把用户体验建立在远端一定成功之上

## 5. iClaw 需要新增的接口能力

### Raw 保存接口

- 保存页面来源
- 保存摘录片段
- 按 source url 查询已有素材
- 拉取当前用户素材列表

## 6. 对 iClaw 客户端的直接价值

一旦接通：

- 插件采集的内容会真实出现在知识库第一栏 Raw tab
- 第二栏能显示素材详情
- 第三栏能以这些真实素材做上下文对话

这会让知识库从纯 UI 原型进入真实对象闭环。
