# iClaw Credit 产品与计费设计

更新时间：2026-03-18

## 1. 目标

将底层多模型、多供应商的 token 成本统一抽象成 iClaw 自有结算单位“龙虾币”。

设计目标：

- 对用户：只暴露一个统一余额单位，降低理解成本。
- 对业务：保留定价权，不把产品层直接绑定到底层 token 价格。
- 对财务：能够回溯真实成本、毛利、免费额度消耗和充值消耗。
- 对风控：免费额度可控，不允许无限囤积与低成本套利。

## 2. 核心原则

### 2.1 不做全局 token:credit 固定比例

不建议定义：

```text
1 token = x 龙虾币
```

原因：

- 不同模型输入/输出价格差异很大。
- 缓存读写、工具调用、搜索、文件解析都不等价于 token。
- 固定比例会让高成本模型亏损，低成本模型定价失真。

建议改为：

```text
真实成本 -> 龙虾币 -> 用户零售价
```

### 2.2 前台显示预估，后台按实际结算

- 发起前：显示“预计消耗”或“预计区间”
- 执行后：按 provider 返回的实际 usage 结算
- 大任务：先预留额度，结束后多退少补

### 2.3 免费额度不做可累积发币

不建议“每天发 200 龙虾币到账户余额，可长期囤积”。

建议做成：

- `daily_free_balance` 每日重置
- 当天不用完即失效
- 不进入长期可累计余额

## 3. 龙虾币单位建议

建议先使用统一内部基准：

```text
1 龙虾币 = $0.0001 成本单位
$1 = 10,000 龙虾币
```

好处：

- 数量级好理解
- 面向用户展示时是整数
- 对后台成本换算足够细

## 4. 账户余额分桶

建议拆成三个用户可消费 bucket，一个系统预留 bucket：

- `daily_free_balance`
  - 每日重置
  - 不累计
  - 优先消耗
- `subscription_balance`
  - 月订阅赠送
  - 可按月失效
- `topup_balance`
  - 用户充值购买
  - 最后消耗
- `reserved_balance`
  - 任务执行中预留
  - 不直接展示为可消费余额

推荐扣减顺序：

1. `daily_free_balance`
2. `subscription_balance`
3. `topup_balance`

## 5. 免费策略建议

### 5.1 每日免费额度

建议：

- 每日重置 `200` 龙虾币
- 登录用户可用
- 不累计
- 仅覆盖标准模型与轻量任务

### 5.2 新用户试用包

建议：

- 注册即送 `1000-3000` 龙虾币试用包
- 7 天失效
- 用于完整体验首个闭环，而不是长期白嫖

### 5.3 免费层限制

- 高成本模型不可用，或以更高倍率扣费
- 超长上下文不可用
- 深度 agent / 批量任务不可用
- 文件/搜索等重成本能力需限频或限场景

## 6. 定价原则

### 6.1 后台成本计算

```text
raw_cost_usd =
  input_tokens * input_price_per_token
  + output_tokens * output_price_per_token
  + cache_read_tokens * cache_read_price_per_token
  + cache_write_tokens * cache_write_price_per_token
  + tool_calls * tool_flat_price
  + other_fixed_costs
```

### 6.2 成本转龙虾币

```text
cost_credits = ceil(raw_cost_usd / 0.0001)
```

### 6.3 用户零售价

```text
charged_credits =
  ceil(cost_credits * markup_multiplier + platform_fee_credits)
```

建议起步：

- `markup_multiplier = 2.5 ~ 4.0`
- `platform_fee_credits = 2 ~ 20`

说明：

- 标准聊天可使用较低倍率
- 高级模型、高工具密度任务可使用更高倍率
- 平台费用于覆盖鉴权、账本、缓存、重试、日志等产品成本

## 7. 预估消耗策略

### 7.1 可精算部分

发请求前可基本估算：

- 系统 prompt
- 历史消息
- 用户输入
- 工具 schema
- 检索结果片段
- 附件提取文本
- 固定工具费用

### 7.2 不可精算部分

发请求前只能统计预测：

- 输出 token
- tool call 次数
- 搜索结果规模
- agent 执行轮数

### 7.3 推荐预测方法

```text
predicted_cost =
  input_tokens_est * input_rate
  + output_tokens_est * output_rate
  + cache_tokens_est * cache_rate
  + tool_calls_est * tool_rate
  + retrieval_est
  + safety_buffer
```

其中：

- 输入 token：直接估算
- 输出 token：按历史统计 profile 取 `p80`
- 展示给用户：使用区间或上限，不使用绝对精确值

### 7.4 前端展示建议

短对话：

- `约 8-20 龙虾币`

较重任务：

- `预计 <= 60 龙虾币`

复杂任务确认弹窗：

- `预计 40-120 龙虾币，本次最多预留 120，完成后按实际结算`

执行完成后：

- `实际消耗 23 龙虾币`

## 8. UI 展示建议

### 8.1 主展示位

建议在 `header` 右上角放余额胶囊：

- `龙虾币 1,240`
- 次级信息：`今日免费 86`

### 8.2 次级入口

头像菜单中增加：

- `龙虾币与账单`

### 8.3 强提示位

发送按钮附近或任务确认弹窗显示：

- 预计消耗
- 免费抵扣量
- 最大预留量

这个展示比单纯余额更重要。

## 9. 低余额与转化

### 9.1 低余额阈值

建议阈值：

- `daily_free_balance < 30`
- 总可用余额 `< 100`

### 9.2 触发动作

- Header 胶囊变色
- 输入框旁提示“余额不足可能影响长任务”
- 弹出轻量充值/升级引导

### 9.3 付费层级建议

可先设计三档：

- Free
  - 每日免费额度
  - 标准模型
- Pro
  - 月赠送额度
  - 允许高级模型
- Max
  - 更高月额度
  - 更高并发与更长任务上限

## 10. 风险与反滥用

- 免费额度必须与账号、设备、风控联动
- 不允许每日免费额度累计
- 免费额度与充值额度必须分桶
- 真实 usage、成本与扣费结果必须全部可回溯
- 长任务要有预留机制，避免跑完再发现余额不足

## 11. 冻结建议

当前阶段建议先冻结以下规则：

- 统一结算单位：龙虾币
- 货币基准：`$1 = 10,000 龙虾币`
- 免费策略：每日重置 `200` 龙虾币
- 余额分桶：`daily_free` / `subscription` / `topup` / `reserved`
- 扣减顺序：免费 -> 订阅 -> 充值
- 前端策略：预估展示，后台实扣
- 长任务策略：先预留，后结算，多退少补
